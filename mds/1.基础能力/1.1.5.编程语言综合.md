## 汇编语言

**是什么？在哪用？**
- **作用**：直接操作CPU和内存的低级语言，提供对硬件的完全控制
- **应用场景**：
  - 对于个别运算量非常大的地方，想做极致的优化
  - 特别是C++的项目，是有机会这么做的
  - 性能关键路径的极致优化
  - 系统编程、驱动开发
  - 嵌入式系统开发

## 知识点

- **寄存器**：CPU中的高速存储单元
- **指令**：CPU执行的基本操作
- **寻址**：访问内存中数据的方式

**会遇到哪些问题？用什么解决？**
- **学习曲线陡峭**
  - **问题**：汇编语言学习难度大，需要深入理解硬件
  - **解决方向**：
    - 从简单的指令开始学习
    - 结合C/C++代码理解汇编
    - 使用编译器生成汇编代码作为参考
    - 只在必要时使用汇编

- **平台依赖性强**
  - **问题**：不同CPU架构的汇编语言不同，代码不可移植
  - **解决方向**：
    - 使用内联汇编（inline assembly）在高级语言中嵌入
    - 为不同平台提供不同的汇编实现
    - 使用条件编译处理平台差异
    - 优先使用高级语言的优化选项

- **维护困难**
  - **问题**：汇编代码可读性差，难以维护
  - **解决方向**：
    - 添加详细的注释
    - 只在关键路径使用汇编
    - 使用内联汇编而非独立汇编文件
    - 提供高级语言的参考实现

**要点和思考方向**
- 汇编语言用于性能关键路径的极致优化
- 学习曲线陡峭，需要深入理解硬件
- 平台依赖性强，代码不可移植
- 维护困难，只在必要时使用
- 结合高级语言使用，而非完全用汇编开发

## 对象

**是什么？在哪用？**
- **作用**：Lua使用表（table）实现面向对象编程
- **应用场景**：需要面向对象特性的Lua脚本

**会遇到哪些问题？用什么解决？**
- **: 和 . 的区别**
  - **问题**：Lua中 `:` 和 `.` 在调用方法时的行为不同
  - **说明**：`:` 会自动传递 `self` 参数，`.` 不会
  - **解决方向**：
    - 对象方法调用使用 `:`，静态方法调用使用 `.`
    - 理解 `obj:method()` 等价于 `obj.method(obj)`
    - 在定义方法时也要使用对应的语法

- **性能问题**
  - **问题**：Lua脚本执行效率相对较低
  - **解决方向**：
    - 关键性能路径用原生代码实现，Lua主要用于业务逻辑
    - 使用LuaJIT提高性能（如果适用）
    - 优化Lua代码，避免不必要的操作
    - 使用C/C++绑定处理性能关键操作

- **调试困难**
  - **问题**：Lua脚本调试工具不如高级语言完善
  - **解决方向**：
    - 使用Lua调试器（如ZeroBrane Studio）
    - 添加日志输出
    - 使用断点和单步调试
    - 提供详细的错误信息

**要点和思考方向**
- Lua适合作为游戏逻辑脚本，支持热更新
- 理解 `:` 和 `.` 的区别，正确使用对象方法
- 关键性能路径用原生代码实现
- 注意Lua与C/C++的交互和绑定

## 表

**是什么？在哪用？**
- **作用**：Lua中的表（table）是唯一的数据结构，可以用来表示数组、字典、对象等
- **应用场景**：所有需要数据结构的地方

**会遇到哪些问题？用什么解决？**
- **表性能问题**
  - **问题**：表操作可能影响性能，特别是大表
  - **解决方向**：
    - 预分配表大小（如果知道大小）
    - 避免频繁的表操作
    - 使用数组部分而非哈希部分（如果适用）

- **表索引混淆**
  - **问题**：数组索引从1开始，容易与C/C++交互时出错
  - **解决方向**：
    - 注意Lua数组索引从1开始
    - 在C/C++交互时进行索引转换
    - 统一使用0或1作为起始索引

**要点和思考方向**
- 表是Lua的唯一数据结构，功能强大
- 注意数组索引从1开始
- 理解表的数组部分和哈希部分

## Java

**是什么？在哪用？**
- **作用**：面向对象的编程语言，具有跨平台特性
- **应用场景**：
  - **服务端开发**：企业级应用开发
  - **Android的原生开发**：Android平台的主要开发语言
  - **工具开发**：比如ant的target等构建工具
  - **大数据处理**：Hadoop、Spark等
  - **企业应用**：Spring框架生态系统

**会遇到哪些问题？用什么解决？**
- **内存管理**
  - **问题**：Java使用GC管理内存，GC暂停可能影响性能
  - **解决方向**：
    - 合理设置JVM参数（堆大小、GC算法）
    - 减少对象创建，使用对象池
    - 避免大对象分配
    - 使用合适的GC算法（G1、ZGC等）

- **性能问题**
  - **问题**：Java性能可能不如C++等编译型语言
  - **解决方向**：
    - JIT编译优化热点代码
    - 使用性能分析工具（如JProfiler）
    - 优化算法和数据结构
    - 使用原生方法（JNI）处理性能关键路径

- **版本兼容性**
  - **问题**：不同Java版本的API和行为可能不同
  - **解决方向**：
    - 明确目标Java版本
    - 使用兼容性测试
    - 注意废弃API的使用

**要点和思考方向**
- Java适合企业级应用开发
- 注意GC对性能的影响，合理配置JVM
- 利用丰富的Java生态（Spring、Hibernate等）
- 理解JVM的工作原理，有助于性能优化

## 并发编程

**是什么？在哪用？**
- **作用**：Go语言的核心特性之一，提供轻量级并发模型
- **应用场景**：需要高并发的服务端应用
- **组件**：
  - **goroutine**：轻量级线程，Go语言的并发执行单元
  - **channel**：用于goroutine之间的通信和同步
  - **条件变量**：用于更复杂的同步场景
  - **同步锁**：互斥锁等同步原语

**会遇到哪些问题？用什么解决？**
- **goroutine泄漏**
  - **问题**：goroutine未正确退出，导致资源泄漏
  - **解决方向**：
    - 使用context控制goroutine生命周期
    - 确保goroutine能够正常退出
    - 使用sync.WaitGroup等待goroutine完成
    - 监控goroutine数量

- **channel阻塞**
  - **问题**：channel操作不当导致死锁或阻塞
  - **解决方向**：
    - 使用带缓冲的channel
    - 使用select处理多个channel
    - 使用超时机制
    - 避免channel死锁

**要点和思考方向**
- goroutine和channel是Go并发编程的核心
- 使用channel进行通信，共享内存通过通信实现
- 注意goroutine泄漏和channel阻塞问题
- 理解Go的并发模型："不要通过共享内存来通信，而要通过通信来共享内存"

## GC（垃圾回收）

**是什么？在哪用？**
- **作用**：自动内存管理，减少内存泄漏风险
- **应用场景**：所有Go程序

**会遇到哪些问题？用什么解决？**
- **GC延迟**
  - **问题**：GC可能导致程序暂停（STW - Stop The World）
  - **解决方向**：
    - 合理控制对象创建，避免频繁分配大对象
    - 使用对象池复用对象
    - 调整GC参数（GOGC环境变量）
    - 使用sync.Pool复用临时对象

- **内存使用过多**
  - **问题**：程序占用内存过多
  - **解决方向**：
    - 减少对象分配
    - 使用值类型而非指针（如果适用）
    - 及时释放不需要的对象
    - 使用内存分析工具（如pprof）

**要点和思考方向**
- Go的GC是并发的，但仍可能影响性能
- 注意GC对性能的影响，合理使用内存
- 使用对象池和sync.Pool减少分配
- 监控GC性能，调整GC参数

## SQL

**是什么？在哪用？**
- **作用**：结构化查询语言，用于数据库操作和查询
- **应用场景**：
  - **后端开发**：数据库操作和查询

  - **问题**：SQL查询性能差，影响系统性能
  - **解决方向**：
    - 优化SQL查询，使用索引
    - 避免N+1查询问题
    - 使用查询分析工具
    - 合理使用缓存

**要点和思考方向**
- SQL是数据库操作的标准语言
- 注意SQL注入安全问题，使用参数化查询
- 优化SQL查询性能，合理使用索引
- 理解数据库事务和隔离级别
