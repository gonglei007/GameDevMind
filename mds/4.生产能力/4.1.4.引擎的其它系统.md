<h2 align="center">引擎的其它系统</h2>
<p>
除了渲染、场景、UI 等「看得见」的部分，引擎内部还有一批支撑性系统，负责资源管理、核心通用能力、跨平台封装以及与第三方 SDK 的衔接。这些系统决定了引擎和产品能否高效开发、稳定运行和长期演进。
</p>

**关键词:**<br/>
*资源管理,核心系统,跨平台接口,第三方SDK,游戏特有系统*

## 资源管理

### 作用

**做什么的？**  
管理所有游戏数字内容在磁盘和运行时中的组织、加载与生命周期。

**核心认识：**
- 所有游戏的数字内容，都要以数据（文件）的形式进行存储和使用
- 资源管理决定了项目的可维护性、加载性能和包体结构

### 典型资源类型

- 3D 模型  
- 贴图  
- 材质  
- 字体  
- 骨骼  
- 碰撞数据  
- 地图 / 关卡数据 等

### 生产视角的要点

- 统一的资源命名与目录规范（方便工具和脚本自动处理）
- 资源依赖关系管理（模型引用的材质、贴图、动画等）
- 支持打 AssetBundle / Pak / 自定义包格式等打包流程

## 核心系统

### 作用

**做什么的？**  
提供所有系统模块都可能用到的通用能力，是引擎的「基础库层」。

### 典型内容

- 内存管理  
- 单元测试框架  
- 数学库（向量、矩阵、四元数、随机数等）  
- 调试与日志系统  
- JSON / XML 等数据解析

### 生产视角的要点

- 统一的日志规范（等级、打点位置、输出格式），为线上问题排查提供基础
- 良好的数学库与工具，减少「各写一套数学函数」的重复劳动
- 单元测试框架的存在，降低引擎和中间件演进的风险

## 跨平台接口

### 作用

**做什么的？**  
对操作系统和底层库进行封装，向上提供统一接口，让引擎逻辑尽量不关心具体平台差异。

### 典型封装方向

- 文件系统  
- 网络  
- 容器 / 线程库  
- 图形接口封装（统一 DirectX / OpenGL / Vulkan / Metal 等差异）  
- 物理接口封装（对不同物理 SDK 的统一抽象）

### 生产视角的要点

- 让上层模块（渲染、物理、资源管理等）只依赖统一接口，可以在项目后期更换/升级底层实现
- 简化多平台发布流程（PC / 主机 / 移动端）的差异处理

## 游戏特有的系统

### 说明

**做什么的？**  
针对不同游戏类型定制的系统，通常建立在引擎通用能力之上。

### 举例

- 武器系统  
- 载具系统  
- 游戏特定的渲染：
  - 水的渲染
  - 地形渲染
- 游戏摄像机：
  - 跟随相机
  - 动画相机
- 角色控制：
  - 状态管理
  - 移动
  - 碰撞

### 思考方向

- 这些系统有明显的「品类特性」，但内部也会沉淀可复用的引擎模块
- 建议在项目后期将通用部分抽象下沉，形成团队级的「游戏中间件」

## 第三方接口 / SDK

### 作用

**做什么的？**  
在操作系统之上，利用成熟的第三方库实现图形、物理、动画等基础功能服务。

### 举例

- 图形接口：
  - DirectX  
  - OpenGL  
  - Vulkan  
- 物理 SDK：
  - PhysX  
  - Havok  
  - ODE  
- 动画 SDK：
  - Granny

### 生产视角的要点

- 通过合理的封装，让第三方 SDK 可被安全升级或替换
- 在选择 SDK 时，综合考虑：性能、稳定性、许可协议、平台支持度与社区活跃度
