<h2 align="center">项目里程</h2>
<p>
一副健康的骨架能确保你顺利的走向终点，前期做出一个清楚的里程规划，能让项目走的更加稳健。项目里程定义了游戏开发过程中的关键节点和阶段，不同类型的游戏（网游、主机游戏）和不同的开发模式（自研、外包）会有不同的里程规划。
</p>

**关键词:**<br/>
*Demo,FPP,Alpha,Beta,封测,内测,公测,Code Freeze,Gold Master*

## 常见问题

### 网游 vs 主机

**问：** 为什么主机游戏和PC网游、手机网游的里程碑不太一样？

**答：** 因为网游可以在游戏上线后持续的更新、维护，但主机游戏的审查机制和迭代更新方式都不同，所以产品需要在发布之前尽量做的完善。

- **网游特点：** 可以持续迭代，上线后可以修复问题、添加内容，里程碑更注重快速验证和迭代
- **主机游戏特点：** 发布后更新困难，需要通过严格的审查，里程碑更注重完整性和稳定性

### 自研 vs 外包

**问：** 自研和外包有什么不同？

**答：**
- **自研：**
  - **以产品和盈利为中心：** 关注产品的市场表现和盈利能力
  - **在有限成本内做出吸引人的产品：** 需要在成本控制下创造有竞争力的产品
- **外包：**
  - **以时间和质量为中心：** 关注按时交付和交付质量
  - **按时、按质的去完成客户的需求：** 严格按照合同要求完成开发任务

## 常规里程

### 立项

**做什么的？** 项目启动阶段，确定项目方向和目标。

**在哪用？** 所有游戏项目的起始阶段。

**会遇到哪些问题？**
- 如何确定项目方向？
- 如何评估项目的可行性？
- 如何平衡创意和市场？

**要点和思考方向：**

**周期最短，意义最重大的一个阶段：** 虽然时间短，但决定了整个项目的方向

**事情：**
- **头脑风暴：** 集思广益，探索各种可能性

**考虑：**
- **确定团队要做什么：**
  - **创作作品，培育市场？** 以创意为核心，探索新的市场空间
  - **研究市场，开发商品？** 以市场为导向，满足现有市场需求
- **确定团队能做什么：**
  - **好点子有没有配套的好执行？团队是否能驾驭？** 评估团队的技术能力和资源，确保项目可执行

### 设计

**做什么的？** 将想法具体化，确定产品的外观、框架和技术方案。

**在哪用？** 立项后的设计阶段。

**会遇到哪些问题？**
- 如何将抽象的想法转化为具体的设计？
- 如何确保团队对产品有统一的理解？
- 如何选择合适的技术方案？

**要点和思考方向：**

**原画：**
- **确定想象的东西，在看到的时候，是不是还是那个感觉：** 通过视觉化验证想法的可行性
- **确保大家想象出来的是同一个东西：** 统一团队对产品的视觉认知
- **从画面中发现之前没有考虑到的事情：** 视觉化过程中可能发现新的问题或机会

**技术选型：**
- **确定产品的设计框架：** 选择合适的设计模式和架构
- **确定产品的技术选型：** 选择合适的技术栈和工具

### 原型/First Playable

**做什么的？** 快速制作可体验的核心玩法版本，验证产品概念。

**在哪用？** 设计阶段之后，正式开发之前。

**会遇到哪些问题？**
- 如何快速验证核心玩法？
- 如何控制原型的成本？
- 何时应该调整方向？

**要点和思考方向：**

- **快速做出来可以体验产品核心内容和玩法的版本，验证最初立项时脑子里的产品的感觉：** 通过实际体验验证想法的可行性
- **这个时候如果要调整方向，代价还不是那么的大：** 在投入大量资源之前，及时调整方向成本较低

### 计划/Planning

**做什么的？** 制定详细的开发计划和时间表。

**在哪用？** 原型验证之后，正式开发之前。

**会遇到哪些问题？**
- 何时制定详细计划？
- 如何平衡计划的详细程度和灵活性？
- 如何应对计划的变化？

**要点和思考方向：**

- **建议这个时候再开始做正式的计划。因为这时候在产品、技术、内容等方面才有了比较清晰的方向：** 在有了原型验证后，对项目的理解更深入，计划更准确
- **当然也有团队一开始就把计划做好了，然后就破釜沉舟了。这就看各个领导者的风格和信心了：** 不同的团队有不同的策略，取决于领导者的风格和项目的特点

## 网络游戏里程

### 开发

**做什么的？** 按设计推进产品功能和内容的实现。

**在哪用？** 原型验证后的正式开发阶段。

### 封测

**做什么的？** 小范围验证产品实际效果，收集反馈。

**在哪用？** 开发到一定阶段后，需要验证产品效果时。

**要点和思考方向：**
- **小范围验证产品实际效果，收集反馈，为产品完善指明方向：** 通过小范围测试发现问题和改进方向

### 内测

**做什么的？** 邀请部分玩家进行多方面的测试，确保游戏质量。

**在哪用？** 封测之后，公测之前。

**要点和思考方向：**
- **在内测阶段，游戏公司邀请一部分玩家对游戏运行性能，游戏设计，游戏平衡性，游戏BUG以及服务器负载等进行多方面测试，以确保游戏在公测后能顺利进行：** 全面测试游戏的各个方面，确保公测的顺利进行
- **内测资料进入公测通常是不保留的，但现在越来越多的游戏公司为了奖励内测玩家，采取公测奖励措施或直接进行不删档内测：** 根据策略决定是否保留内测数据

### 公测

**做什么的？** 面向所有玩家开放测试。

**在哪用？** 内测验证之后，正式上线之前。

**要点和思考方向：**
- **内测资料进入公测通常是不保留的，但现在越来越多的游戏公司为了奖励内测玩家，采取公测奖励措施或直接进行不删档内测：** 平衡玩家体验和游戏公平性

### 上线

**做什么的？** 正式发布游戏，开始运营。

**在哪用？** 公测验证之后。

**要点和思考方向：**
- **导量、推广，运营：** 上线后需要持续的运营和推广
- **一切才刚刚开始！！！** 上线不是结束，而是持续运营的开始

## 主机游戏里程

### Alpha

**做什么的？** 玩法、功能和内容基本完整，可以充分体验游戏。

**在哪用？** 开发到一定阶段，核心内容基本完成时。

**要点和思考方向：**
- **玩法、功能和内容基本完整了。可以充分的体验游戏了：** 游戏的核心体验已经完整

### Code Freeze

**做什么的？** 代码冻结，停止新功能开发，专注于修复问题。

**在哪用？** Alpha之后，Beta之前。

### Beta

**做什么的？** 充分测试、修复问题，获得稳定的准备发布的版本。

**在哪用？** Code Freeze之后，Release之前。

**要点和思考方向：**
- **充分的测试、修复问题。获得一个稳定的，准备发布的版本：** 通过大量测试确保版本稳定

### Code release

**做什么的？** 代码发布版本，准备进入最终阶段。

**在哪用？** Beta测试通过后。

### Gold master

**做什么的？** 最终发布版本，准备提交审查。

**在哪用？** Code release之后，正式发布之前。

### Release

**做什么的？** 正式发布，上线赚钱。

**在哪用？** 通过审查后，正式发布。

**要点和思考方向：**
- **正式发布，上线赚钱了：** 主机游戏发布后更新困难，需要确保版本质量
