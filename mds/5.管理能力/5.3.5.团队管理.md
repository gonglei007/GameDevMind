<h2 align="center">团队管理</h2>
<p>
真正推动公司和项目前进的是团队中的人，每个人的方向正确、驱动力充足，就会让事情事半功倍。团队管理包括验收标准定义、问题发现、需求理解、目标对齐、利益协调等方面，是保证团队高效协作的关键。
</p>

**关键词:**<br/>
*团队管理,验收标准,目标对齐,利益共同点,问题发现*

**标签:**<br/>
*等级: 中级, 阶段: 开发, 分类: 管理能力, 角色: 管理*

## 常见问题与解决方案

### 一、结点验收标准的定义

**是什么？在哪用？**

- **作用**：定义工作节点的验收标准，确保交付质量
- **应用场景**：
  - 所有需要阶段性交付的项目
  - 工作节点验收时
  - 需要确保交付质量时
- **做什么的？** 定义工作节点的验收标准，确保交付质量。
- **在哪用？** 所有需要阶段性交付的项目。

**会遇到哪些问题？用什么解决？**

- **如何定义验收标准？**
  - **问题**：需要定义验收标准
  - **解决方向**：
    - **问题：**
      - **以被分配的任务或者功能点为导向。每个环节都认为已经完成了，这个事情就完成了。结果一个周期下来，所做的事情还是不可见、不可审查、或者问题多多的东西：只关注任务完成，不关注实际效果：** 任务导向问题
        - 以任务为导向
        - 不关注实际效果
        - 不可见、不可审查
    - **解决：**
      - **以体验为导向，以用例为导向。在阶段过程中，团队共同审视是否达成体验目标、是否达成用例效果：关注实际效果，而非仅仅完成任务：** 体验导向
        - 以体验为导向
        - 以用例为导向
        - 达成体验目标
        - 达成用例效果
    - 以体验为导向定义标准
    - 以用例为导向定义标准
    - 团队共同审视
    - 明确体验目标和用例效果

- **如何确保交付质量？**
  - **问题**：需要确保交付质量
  - **解决方向**：
    - 建立明确的验收标准
    - 在阶段过程中持续审视
    - 团队共同验收
    - 及时反馈和改进

- **如何避免"完成了但不可用"的情况？**
  - **问题**：需要避免"完成了但不可用"的情况
  - **解决方向**：
    - 以体验和用例为导向
    - 确保可见和可审查
    - 在过程中持续审视
    - 及时发现问题并解决

**要点和思考方向：**
- 验收标准应该以体验为导向，以用例为导向
- 在阶段过程中团队共同审视是否达成体验目标和用例效果
- 避免只关注任务完成，不关注实际效果
- 确保交付内容可见、可审查、可用

### 二、问题到很后期才去反应

**是什么？在哪用？**

- **作用**：及时发现和解决问题，避免问题积累
- **应用场景**：
  - 整个项目开发过程
  - 需要持续监控质量时
  - 需要避免问题积累时
- **做什么的？** 及时发现和解决问题，避免问题积累。
- **在哪用？** 整个项目开发过程。

**会遇到哪些问题？用什么解决？**

- **如何及时发现问题？**
  - **问题**：需要及时发现问题
  - **解决方向**：
    - **问题：**
      - **日常开发只关注在时间和事项上，不去发现问题。结果开发了一两个月了，才去审视问题，才发现有很多难以挽回的结果：只关注进度，不关注质量：** 进度导向问题
        - 只关注时间和事项
        - 不去发现问题
        - 问题积累到后期
        - 难以挽回
    - **解决：**
      - **日常持续发现问题。比如：技术负责人在前期设计就要接入，在日常开发就要Review，团队没有以目标为导向就要指出。把所有问题解决到过程之中：在过程中持续发现问题并解决，而非等到最后：** 持续发现问题
        - 日常持续发现问题
        - 前期设计接入
        - 日常开发Review
        - 指出目标导向问题
        - 在过程中解决问题
    - 技术负责人在前期设计接入
    - 日常开发进行Review
    - 指出目标导向问题
    - 建立问题发现机制

- **如何避免问题积累？**
  - **问题**：需要避免问题积累
  - **解决方向**：
    - 在过程中持续发现问题
    - 及时解决问题
    - 建立问题跟踪机制
    - 定期审查问题

- **如何建立问题发现机制？**
  - **问题**：需要建立问题发现机制
  - **解决方向**：
    - 建立代码审查机制
    - 建立设计审查机制
    - 建立定期检查机制
    - 建立问题反馈机制

**要点和思考方向：**
- 日常持续发现问题，而非等到最后
- 技术负责人在前期设计就要接入，在日常开发就要Review
- 指出团队没有以目标为导向的问题
- 把所有问题解决到过程之中，避免问题积累

### 三、混淆需求和思路

**是什么？在哪用？**

- **作用**：区分需求（要什么）和思路（怎么做），确保理解正确
- **应用场景**：
  - 需求沟通时
  - 任务分配时
  - 需要明确需求时
- **做什么的？** 区分需求（要什么）和思路（怎么做），确保理解正确。
- **在哪用？** 需求沟通和任务分配时。

**会遇到哪些问题？用什么解决？**

- **如何区分需求和实现思路？**
  - **问题**：需要区分需求和实现思路
  - **解决方向**：
    - **问题：**
      - **有的时候，策划会把一个系统实现思路当成功能需求给到开发人员。有些经验少的开发人员，可能就会把这个当成了他要去执行的任务，结果做出来了一个实际上是由策划设计的技术实现方案：将实现思路误认为是需求：** 混淆问题
        - 实现思路当成需求
        - 误认为是任务
        - 做出技术实现方案
    - **解决：**
      - **尽量去挖掘对方的根本需求，比如想要有的体验、想要解决的问题等，而不是想要实现的方法：挖掘真实需求，而非直接采用对方提出的实现方案：** 挖掘需求
        - 挖掘根本需求
        - 关注体验
        - 关注要解决的问题
        - 而非实现方法
    - 区分"要什么"和"怎么做"
    - 挖掘根本需求
    - 关注体验和效果

- **如何挖掘真实需求？**
  - **问题**：需要挖掘真实需求
  - **解决方向**：
    - **举例：**
      - **策划说要一个匹配系统，他可以说这个匹配的过程、逻辑和流程，但我们要挖掘的是他想要的匹配的体验和达到的效果。如果他的方案可以达成他的根本目的就采纳，如果不能，就跟他去挖掘这个要达成的目的：关注体验和效果，而非具体的实现方式：** 需求挖掘示例
        - 挖掘匹配的体验
        - 挖掘达到的效果
        - 关注根本目的
        - 而非具体实现方式
    - 询问"为什么"而非"怎么做"
    - 关注体验和效果
    - 理解根本目的
    - 评估方案是否达成目的

- **如何避免按错误思路实现？**
  - **问题**：需要避免按错误思路实现
  - **解决方向**：
    - 先明确真实需求
    - 评估实现思路是否合适
    - 与需求方确认理解
    - 选择最佳实现方案

**要点和思考方向：**
- 区分需求（要什么）和思路（怎么做）
- 挖掘对方的根本需求，关注体验和要解决的问题
- 关注体验和效果，而非具体的实现方式
- 如果方案可以达成根本目的就采纳，否则挖掘真实目的

### 四、过度追求"速度"

**是什么？在哪用？**

- **作用**：平衡开发速度和代码质量，避免技术债务
- **应用场景**：
  - 功能开发时
  - 系统设计时
  - 需要平衡速度和质量时
- **做什么的？** 平衡开发速度和代码质量，避免技术债务。
- **在哪用？** 功能开发和系统设计时。

**会遇到哪些问题？用什么解决？**

- **如何平衡速度和质量？**
  - **问题**：需要平衡速度和质量
  - **解决方向**：
    - **问题：**
      - **有些开发人员很容易过度沉迷于看上去最快速的完成任务，而完全忽略掉这个任务最终是要达成什么目标、这个系统是否有可维护性、可扩展性。使得问题暴露于后期，又很难调整或维护，极端情况甚至要推翻重做：只关注速度，忽略质量和可维护性：** 速度导向问题
        - 过度追求速度
        - 忽略目标
        - 忽略可维护性
        - 忽略可扩展性
        - 问题暴露于后期
    - **解决：**
      - **在项目计划和功能设计的时候多参与、多沟通，过程多跟进。尽量表述好这个阶段任务的目的、目标。去达成目标、而不是去完成那一行文字所表达的任务：明确目标，关注质量，而非仅仅完成任务：** 目标导向
        - 多参与、多沟通
        - 过程多跟进
        - 表述任务目的和目标
        - 达成目标
        - 而非完成任务
    - 明确任务目的和目标
    - 关注可维护性和可扩展性
    - 在过程中多跟进
    - 平衡速度和质量

- **如何避免过度追求速度？**
  - **问题**：需要避免过度追求速度
  - **解决方向**：
    - 明确任务目标
    - 关注系统质量
    - 评估技术债务
    - 建立质量检查机制

- **如何评估技术债务？**
  - **问题**：需要评估技术债务
  - **解决方向**：
    - 评估可维护性
    - 评估可扩展性
    - 评估重构成本
    - 建立技术债务管理机制

**要点和思考方向：**
- 避免过度追求速度，忽略质量和可维护性
- 在项目计划和功能设计时多参与、多沟通
- 明确任务的目的和目标，达成目标而非仅仅完成任务
- 关注系统的可维护性和可扩展性

### 五、寻找利益共同点

**是什么？在哪用？**

- **作用**：找到公司和个人的共同利益，促进合作共赢
- **应用场景**：
  - 团队管理时
  - 激励团队成员时
  - 需要促进合作时
- **做什么的？** 找到公司和个人的共同利益，促进合作共赢。
- **在哪用？** 团队管理和激励时。

**会遇到哪些问题？用什么解决？**

- **如何找到共同利益？**
  - **问题**：需要找到共同利益
  - **解决方向**：
    - **问题：**
      - **公司是个平台，团队中的每个人是推动力。有些人会很纯粹的把公司与个人的关系看成了互动的甲方乙方的关系，这可能会使得合作共赢的关系变成互相应对的关系，削弱了团队的主观能动性：将公司和个人关系视为对立：** 对立关系问题
        - 视为甲方乙方关系
        - 互相应对
        - 削弱主观能动性
    - **解决：**
      - **尽量找到各方的共同利益点——通常就是项目的成功。作为管理者，一方面去成就公司的大小目标和利益，一方面去成就团队中的每个人的发展和利益：寻找共同利益，实现双赢：** 共同利益
        - 找到共同利益点
        - 项目成功
        - 成就公司目标和利益
        - 成就个人发展和利益
    - 寻找项目成功等共同利益点
    - 成就公司目标和利益
    - 成就个人发展和利益

- **如何平衡公司和个人利益？**
  - **问题**：需要平衡公司和个人利益
  - **解决方向**：
    - **举例：** 共同利益示例
      - **赚钱的共同利益。项目做的更好。团队就有机会获得更多的奖励；公司就会获得更丰厚的利润：项目成功带来双赢：** 赚钱共同利益
        - 项目做得更好
        - 团队获得奖励
        - 公司获得利润
      - **技术眼界的共同利益。项目做的更大、更优质，团队成员就会有机会接触更高价值的技术成长；公司就会获得更扎实的技术储备：技术成长带来双赢：** 技术成长共同利益
        - 项目更大、更优质
        - 个人技术成长
        - 公司技术储备
      - **成长的共同利益。公司通过更优的技术能力、服务或产品，得到成长；个人也就有机会获得更大的成长空间：共同成长带来双赢：** 成长共同利益
        - 公司成长
        - 个人成长空间
    - 寻找双赢的共同利益点
    - 平衡短期和长期利益
    - 建立激励机制

- **如何避免对立关系？**
  - **问题**：需要避免对立关系
  - **解决方向**：
    - 寻找共同利益点
    - 建立合作共赢关系
    - 避免甲方乙方思维
    - 促进团队主观能动性

**要点和思考方向：**
- 寻找公司和个人的共同利益点，通常就是项目的成功
- 作为管理者，一方面成就公司目标和利益，一方面成就个人发展和利益
- 项目成功、技术成长、共同成长等都是双赢的共同利益点
- 避免将公司和个人关系视为对立，促进合作共赢

### 六、一致目标

**是什么？在哪用？**

- **作用**：确保团队成员对目标有一致的理解
- **应用场景**：
  - 任务分配时
  - 目标设定时
  - 需要团队协作时
- **做什么的？** 确保团队成员对目标有一致的理解。
- **在哪用？** 任务分配和目标设定时。

**会遇到哪些问题？用什么解决？**

- **如何确保目标一致？**
  - **问题**：需要确保目标一致
  - **解决方向**：
    - **问题：**
      - **公司每个人都有不同甚至南辕北辙的目标，如果在每个事情上，不跟团队成员沟通好团队目标，大家的出发点就可能会千差万别。有的人的目标可能是"这事我要赶快做完"，有的人的目标可能是"这事我要用最牛的技术"，有的人的目标可能是"这事我要一劳永逸的干掉它"：个人目标不一致导致效率低下：** 目标不一致问题
        - 个人目标不同
        - 出发点千差万别
        - 效率低下
    - **解决：**
      - **大家有这不同的目标，会让一个团队事倍功半。但项目在不同的阶段、不同的条件下，每一个事情的真实目标可能各不相同。这就需要挖掘出来这个阶段这个事情的真实目标，并且，跟每一个相关人员沟通清楚这个事情的真实目标：明确并沟通真实目标，确保团队目标一致：** 目标对齐
        - 挖掘真实目标
        - 沟通清楚真实目标
        - 确保团队目标一致
    - 挖掘每个阶段的真实目标
    - 与每个相关人员沟通清楚
    - 确保团队目标一致

- **如何沟通目标？**
  - **问题**：需要沟通目标
  - **解决方向**：
    - 明确表述真实目标
    - 与相关人员沟通
    - 确认理解一致
    - 定期回顾和调整

- **如何处理目标冲突？**
  - **问题**：需要处理目标冲突
  - **解决方向**：
    - 挖掘真实目标
    - 寻找共同目标
    - 平衡不同目标
    - 建立目标优先级

**要点和思考方向：**
- 个人目标不一致会导致团队事倍功半
- 项目在不同阶段、不同条件下，每个事情的真实目标可能不同
- 需要挖掘出每个阶段每个事情的真实目标
- 与每个相关人员沟通清楚真实目标，确保团队目标一致

### 七、"追责"这个事

**是什么？在哪用？**

- **作用**：建立问题处理机制，避免过度追责导致的问题
- **应用场景**：
  - 问题处理时
  - 团队管理时
  - 需要建立健康团队文化时
- **做什么的？** 建立问题处理机制，避免过度追责导致的问题。
- **在哪用？** 问题处理和团队管理时。

**会遇到哪些问题？用什么解决？**

- **如何处理问题？**
  - **问题**：需要处理问题
  - **解决方向**：
    - **问题：**
      - **为什么团队中总有人遇到问题的第一反应是"甩锅"？为什么团队成员之间不是互相配合而是互相内斗？过度追责导致的问题：** 追责问题
        - 第一反应是甩锅
        - 互相内斗
        - 过度追责导致
    - **解决：**
      - **适度的压力，可以形成动力。过度的压力，就是内部损耗的原动力了：适度压力有益，过度压力有害：** 压力管理
        - 适度压力有益
        - 过度压力有害
      - **在项目过程中，遇到了某个问题，给团队成员的第一信号，不要是指责，而是先让相关的人凑一起，大家优先看看这个问题当下怎么解决，将来怎么规避：优先解决问题，而非追责：** 问题处理
        - 不要是指责
        - 相关人凑一起
        - 优先解决问题
        - 将来怎么规避
      - 优先解决问题，而非追责
      - 让相关人员一起讨论
      - 关注当下解决和将来规避

- **如何避免甩锅？**
  - **问题**：需要避免甩锅
  - **解决方向**：
